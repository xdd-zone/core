# 后台底座优先 V1 实现文档

## 1. 实现目标

按“后台底座优先”方案，对现有认证、用户、RBAC 进行一次收口式实现，直接删除不再需要的旧能力，不保留兼容逻辑。

## 2. 实现原则

1. 先删后收口，再补文档与测试。
2. 旧接口、旧权限、旧文案不保留兼容层。
3. 所有实现以最小权限集和固定角色集为准。
4. 身份域和访问控制域边界必须在代码层落地。

## 3. 实现阶段

### 阶段一：模型与语义清理

目标：

- 收口模型语义
- 移除无效字段语义
- 明确用户生命周期

任务：

- 直接移除 `Role.parentId` 与 `Role.level`
- 调整 `User` 删除语义，禁止默认硬删除入口
- 确认 `deletedAt`、状态字段、查询逻辑一致
- 补齐 `assignedBy` 的真实写入语义

产出：

- 收口后的 Prisma schema
- 对应 service / repository 逻辑同步更新

### 阶段二：权限常量与 seed 收口

目标：

- 只保留最小权限集
- 固定系统角色权限映射

任务：

- 清理旧的权限常量
- 清理未实际使用或不再允许的权限项
- 重写权限 seed
- 重写角色与权限映射 seed
- 为 `superAdmin` 改为显式全权限语义

产出：

- 收口后的权限常量
- 收口后的 seed 脚本

### 阶段三：RBAC 接口清理

目标：

- 将 RBAC 从“动态权限平台”收口为“后台授权能力”

任务：

- 移除角色 CRUD 路由
- 移除角色继承管理路由
- 移除权限 CRUD 路由
- 移除角色权限动态编辑路由
- 保留用户角色分配、用户权限查看、角色列表等必要接口
- 保留当前用户角色、当前用户权限查询
- 同步清理 OpenAPI 文案和响应说明

产出：

- 收口后的 RBAC route / contract / service

### 阶段四：用户模块收口

目标：

- 将用户模块收口为平台资料与后台管理能力

任务：

- 明确“自己”和“管理员”的查询、更新能力
- 去掉与硬删除默认语义不一致的实现
- 调整文案为“停用/封禁/归档”优先
- 保证用户列表、详情、更新接口的权限语义稳定
- 显式保留 `/api/user/me` 与 `/api/user/:id/status`

产出：

- 收口后的 user route / service / repository

### 阶段五：鉴权与权限插件收口

目标：

- 明确 `permission`、`own`、`me` 的使用边界

任务：

- 将 `own` 限定为用户自己资料场景
- 清理对未来业务资源不适用的“伪通用 own 语义”
- 为 `superAdmin` 增加稳定的全权限通路
- 去掉与角色继承相关的缓存/失效复杂逻辑

产出：

- 收口后的 access control 插件

### 阶段六：文档、文案、测试同步

目标：

- 保证对外描述和实际实现一致

任务：

- 清理旧 README、RBAC 文档、OpenAPI 文案
- 清理旧测试用例中的废弃语义
- 为新权限集与新接口边界补测试
- 更新设计、实现、验收文档

产出：

- 最新文档
- 最新测试基线

## 4. 代码清理清单

本次需要直接移除的旧内容包括：

- 角色继承相关实现
- 动态权限管理相关路由和服务
- 角色权限动态编辑相关实现
- 与最小权限集冲突的旧权限常量
- 与硬删除默认语义相关的旧实现
- 与旧设计不一致的文案和注释

## 5. 文件层建议

优先涉及的目录：

- `packages/nexus/src/infra/database/prisma/schema`
- `packages/nexus/src/infra/database/prisma/seed`
- `packages/nexus/src/core/access-control`
- `packages/nexus/src/core/permissions`
- `packages/nexus/src/modules/user`
- `packages/nexus/src/modules/rbac`
- `packages/nexus/src/routes`
- `docs/`

## 6. 实现顺序建议

推荐按以下顺序推进：

1. Schema 与权限常量先收口
2. Seed 与 service / repository 同步收口
3. Route 与 contract 清理
4. 文档、OpenAPI、测试同步

## 7. 完成定义

满足以下条件，视为实现完成：

1. 代码中不再保留本次明确废弃的旧能力。
2. 权限、角色、路由、文档语义一致。
3. 超级管理员能力稳定，不依赖旧权限快照。
4. 用户生命周期实现与文案一致。
5. 测试与验收清单可直接验证本次收口结果。
